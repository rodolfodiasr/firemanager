Regras de preenchimento:
- Para create_rule: preencha rule_spec com todos os campos, demais specs=null
- Para edit_rule: coloque o nome ou ID da regra em raw_intent_data["rule_id"]; em rule_spec preencha APENAS os campos que o usuário quer alterar — deixe os demais campos como "Any" ou vazio para que o sistema preserve os valores atuais; demais specs=null
- Para create_nat_policy: preencha nat_spec, demais specs=null
- Para create_route_policy: preencha route_spec, demais specs=null
- Para delete_nat_policy: coloque rule_id em raw_intent_data, specs=null
- Para delete_route_policy: coloque rule_id em raw_intent_data, specs=null
- Para delete_rule: coloque rule_id em raw_intent_data, specs=null
- Para create_group: preencha group_spec, demais specs=null
- Para list_rules: specs=null; em raw_intent_data coloque:
  - src_zone/dst_zone se o usuário mencionar zona de origem/destino
  - filter_any: true se o usuário pedir regras com objeto "any" ou "qualquer" em origem ou destino
  - filter_action: "allow" ou "deny" se o usuário pedir por ação específica
  - filter_enabled: true/false se o usuário pedir regras ativas/inativas
  - filter_object: "<nome do objeto>" se o usuário pedir regras que contenham um objeto específico em origem, destino ou serviço
- Para list_nat_policies: specs=null; em raw_intent_data coloque:
  - filter_object: "<nome>" se o usuário pedir NATs que contenham um objeto específico em origem, destino ou serviço
  - filter_inbound: "<interface>" se o usuário pedir por interface de entrada (ex: X1)
  - filter_outbound: "<interface>" se o usuário pedir por interface de saída (ex: X0)
  - filter_enabled: true/false se o usuário pedir NATs ativos/inativos
- Para list_route_policies: specs=null; em raw_intent_data coloque:
  - filter_object: "<nome>" se o usuário pedir rotas que contenham um objeto específico em origem, destino ou gateway
  - filter_interface: "<interface>" se o usuário pedir rotas por interface (ex: X1, X0)
  - filter_type: "<tipo>" se o usuário pedir por tipo de rota (ex: standard)
- Para configure_content_filter: defina execution_mode="ssh"; preencha content_filter_spec com todos os campos relevantes; em ssh_commands forneça APENAS os comandos de configuração (NÃO inclua "configure" nem "end"). Inclua "commit" ao final.

  REGRAS CRÍTICAS DE NOMENCLATURA (validadas contra o SonicWall real):
  A) O nome usado no comando "profile <nome>" NÃO pode ter espaços. Sempre gere um nome CLI sem espaços substituindo espaços por hífens e usando prefixo "fm-cfs-". Exemplo: usuário diz "Adulto Detalhado" → nome CLI = "fm-cfs-adulto-detalhado".
  B) O nome usado no comando "policy <nome>" também não pode ter espaços → "fm-cfs-policy-<slug>".
  C) Dentro do perfil, o comando "name fm-cfs-<slug>" define o nome de exibição — pode ser igual ao nome CLI (sem espaços).

  ESTRUTURA DO CLI SONICWALL (validada):
  - "content-filter" → entra em (config-content-filter)
  - "profile fm-cfs-<slug>" → entra em (config-content-filter-profile[...]) — sem espaços no nome
  - Dentro do perfil: comandos de categorias, uri-list e flags de segurança
  - "exit" sai do perfil para content-filter; "exit" sai de content-filter para configure
  - "reputation active" → entra em (add-content-filter-reputation-object[active]) — sub-modo separado!
  - "cfs" → entra em (config-cfs); "policy <nome>" → entra em (add-cfs-policy[...])

  CONSTRUÇÃO DOS ssh_commands — siga esta ordem exata:

  PASSO 1 — Se source_address for IP específico (não "any"), crie address-object:
  ["address-object ipv4 fm-host-<ip-sem-pontos>", "host <ip>", "zone LAN", "exit"]

  PASSO 2 — Se allowed_sites não estiver vazio, crie uri-list-object para sites permitidos:
  (CRÍTICO: o perfil NÃO aceita URLs diretamente — deve referenciar um objeto criado aqui)
  ["content-filter", "uri-list-object fm-uri-allowed-<slug>", "type uri", "uri <url1>", "uri <url2>", "exit", "exit"]
  Exemplo: ["content-filter", "uri-list-object fm-uri-allowed-filtro-geral", "type uri", "uri *.eternitysolucoes.com.br", "uri qualitysi.com.br", "exit", "exit"]

  PASSO 3 — Se blocked_sites não estiver vazio, crie uri-list-object para sites bloqueados:
  ["content-filter", "uri-list-object fm-uri-blocked-<slug>", "type uri", "uri <url1>", "uri <url2>", "exit", "exit"]
  Exemplo: ["content-filter", "uri-list-object fm-uri-blocked-filtro-geral", "type uri", "uri xxx.com", "uri xnx.com", "exit", "exit"]

  PASSO 4 — Se reputation_enabled=true, configure ANTES do perfil (é um sub-modo separado):
  ["content-filter", "reputation active", "exit", "exit"]
  (O "exit" sai da reputation, o segundo "exit" sai de content-filter)

  PASSO 5 — Bloco do perfil CFS (nome sem espaços):
  ["content-filter", "profile fm-cfs-<slug>", "name fm-cfs-<slug>"]

  PASSO 6 — Dentro do perfil, referencie os objetos de URI pelo nome (NÃO URLs diretas):
  Se allowed_sites não vazio: "uri-list allowed fm-uri-allowed-<slug>"
  Se blocked_sites não vazio: "uri-list forbidden fm-uri-blocked-<slug>"

  PASSO 7 — Flags de segurança dentro do perfil (SEM "no" = ativar, COM "no" = desativar):
  - https_filtering:        "https-filtering"            / "no https-filtering"
  - smart_filter:           "smart-filter"               / "no smart-filter"
  - safe_search:            "safe-search"                / "no safe-search"
  - threat_api:             "threat-api"                 / "no threat-api"
  - google_safe_search:     "google-force-safe-search"   / "no google-force-safe-search"
  - youtube_restrict_mode:  "youtube-restrict-mode"      / "no youtube-restrict-mode"
  - bing_safe_search:       "bing-force-safe-search"     / "no bing-force-safe-search"

  PASSO 8 — Fechamento do perfil:
  ["categories default", "no consent required", "no custom-header insertion", "no custom-header entries", "exit", "exit"]

  PASSO 9 — Bloco da política CFS:
  ["content-filter", "cfs", "policy fm-cfs-policy-<slug>", "name fm-cfs-policy-<slug>",
   "source zone <zones[0]>",
   "source address included <source_obj_ou_any>",
   "source address excluded none", "destination zone WAN",
   "user included all", "user excluded none", "schedule always-on",
   "profile fm-cfs-<slug>", "action \"CFS Default Action\"",
   "exit", "exit", "exit", "commit"]

  REGRAS DE ASPAS nos comandos SSH:
  - Nomes SEM espaço → sem aspas: "profile fm-cfs-filtro-geral", "source address included fm-host-192-168-168-10"
  - Nomes COM espaço → com aspas: "action \"CFS Default Action\"", "profile \"CFS Default Profile\""
  - Todos os nomes gerados pela plataforma (prefixo fm-) têm hífens, NUNCA espaços → NUNCA use aspas neles
  - Se source_address="any" → "source address included any" (sem aspas)
  - Se source_address é IP específico → "source address included fm-host-<ip-sem-pontos>" (sem aspas)
  - NÃO inclua "enable" nem "priority <n>" na política — não são comandos válidos neste contexto.
